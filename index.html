I'm an AI Systems Engineer passionate about designing intelligent software systems that solve real-world business problems.

Engineering Principles

The core philosophies that guide my architectural choices, system designs, and engineering decisions.

01

Start with the Business Problem

Start with the business problem before choosing technology. Technology is a tool to solve business needs, not the target. Every system design begins by identifying core constraints first.

02

Architecture Before Implementation

Architecture is more important than implementation. A clean design, modular boundaries, and structured information flows survive code refactoring and model upgrades.

03

Build Simple, Then Iterate

Build the simplest working solution first. Avoid premature optimizations and unnecessary system complexity, then continuously iterate based on concrete evidence.

04

AI is Part of a Larger Software System

AI is one component of a complete software system. True reliability, security, and performance depend on strong backend engineering, input validation, and fallback handlers.

05

Every Trade-Off Should Be Intentional

Engineering is the discipline of managing trade-offs. Decisions regarding cost, latency, throughput, complexity, and accuracy must be made consciously and documented.

06

Learn Through Building

Deep understanding comes from writing code, breaking systems, and debugging failures. Theoretical knowledge is validated only through hands-on system building.

07

Never Treat Technology as a Black Box

Avoid treating tools, libraries, or AI models as black boxes. Dig into how components behave under the hood to know why they fail, what trade-offs they introduce, and how they scale.

08

Think in Systems

Think in systems rather than isolated technologies. Analyze how components interact, how data flows across boundaries, and the downstream effects of upstream changes.

Engineering Journey

Transitioning from software development to AI Systems Engineering.

5 Months • AI Solutions Team

AI/ML Intern

L&T Technology Services (LTTS)

Experience

Overview

My internship at LTTS was where I transitioned from building software to engineering AI systems. Unlike traditional internship experiences where tasks are broken into implementation tickets, I was often presented with business problems rather than predefined solutions. The expectation wasn't simply to write code—it was to research, evaluate, architect, prototype, and validate AI-driven approaches that could solve real enterprise challenges.

"This environment fundamentally changed how I approach engineering. It taught me that successful software begins with understanding the problem rather than immediately choosing the technology."

Engineering Mindset Developed

The most valuable lesson from LTTS wasn't learning Python; it was learning how to approach ambiguous engineering problems. That shift fundamentally changed how I think as an engineer.

Instead of asking:
"What code should I write?"
I learned to ask:
  • What problem are we solving?
  • Why is the current solution insufficient?
  • What architecture best fits the constraints?
  • How should the system evolve over time?

What This Experience Represents

This internship should not be viewed as five months of Python development. It represents the point where my engineering focus shifted toward designing intelligent software systems. It reinforced three engineering principles that continue to guide my work today:

Principle 1
Start with the business problem.
Principle 2
Think in systems rather than isolated components.
Principle 3
Build solutions that remain maintainable long after deployment.
"Looking back, this internship shaped me far beyond the technologies I learned. It changed how I approach engineering. Today, whenever I begin a new project, I no longer start by asking which framework or model to use. Instead, I start by understanding the business problem, identifying constraints, and designing a system capable of solving it in a way that is scalable, maintainable, and practical. That shift—from writing software to architecting systems—is the most valuable outcome of my experience at LTTS."

Architectural Ownership

One of the defining characteristics of my internship was ownership. Instead of receiving detailed instructions, I was given a business problem and expected to determine:

  • ✦ Whether AI was the appropriate solution.
  • ✦ Which architecture would best address the requirement.
  • ✦ What technologies should be used.
  • ✦ How the overall system should be structured.
  • ✦ How different components should interact.
  • ✦ What trade-offs existed.
  • ✦ How the solution could be validated.

This level of ownership pushed me beyond implementation and into architectural decision-making.

Active Learning & Research

Current Focus

Advanced AI Systems

Designing robust, multi-component architectures that orchestrate intelligence, combining semantic retrieval, multi-modal inputs, and structured output validation.

Why it matters

Isolated models are simple API calls; enterprise value is created when models are integrated into reliable, self-correcting systems.

Currently Investigating

Advanced evaluation frameworks (Ragas, Phoenix) to benchmark retrieval accuracy and semantic drift.

Agentic AI

Building autonomous execution loops where models act as planners, leverage tools dynamically, and coordinate within multi-agent teams.

Why it matters

Agentic workflows allow software to self-correct, call external APIs, and execute complex workflows without constant human supervision.

Currently Investigating

Robust guardrails for agent loops to prevent execution state drift, infinite loops, and API rate-limit exhaustion.

Model Adaptation

Modifying existing pre-trained open models to behave reliably on domain-specific structures and rules using parameter-efficient fine-tuning (PEFT).

Why it matters

General-purpose models struggle with proprietary schemas. Targeted fine-tuning achieves high accuracy at a fraction of the compute and inference cost.

Currently Investigating

Direct Preference Optimization (DPO) and ORPO to align model reasoning styles directly on custom datasets.

Production AI Engineering

Optimizing LLM inference pipelines, cost structures, throughput, latency, and token consumption for enterprise-scale deployments.

Why it matters

A great prototype fails in production if latency is too high or token cost scales linearly with usage.

Currently Investigating

Speculative decoding, semantic caching layers, and custom token-rate limiting controllers in API gateways.

Enterprise Software Architecture

Designing decoupled, event-driven backend systems that handle asynchronous workflows and manage state transitions cleanly.

Why it matters

AI components are inherently non-deterministic. Wrapping them in robust, deterministic software architectures guarantees system stability.

Currently Investigating

Building distributed event queues (RabbitMQ, Kafka) optimized for streaming long-running model outputs.
